Final Summary Explanation:
2. The Architectural Analysis: Why Java redlined your CPU
The most staggering piece of data you collected here is the CPU utilization. Java hit 772% CPU, essentially maxing out 8 of your CPU's threads, while Go only needed 257% CPU to do more work.

Why did this happen? It comes down to how the servers handle concurrency:

Java's "Thread-Per-Request" Model: By default, Spring Boot uses Tomcat. Tomcat assigns a heavy, operating-system-level Thread to every single user. When 100 users hit the API simultaneously, Tomcat instantly spun up 100 massive OS threads. Your CPU spent a massive amount of its energy just rapidly switching between these 100 threads (this is called "Context Switching") instead of actually processing the JSON payload. That is why the CPU skyrocketed and the latency slowed down to 3.3 seconds.

Go's "Goroutine" Model: Go does not use heavy OS threads. It uses lightweight "Goroutines." When 100 users hit the Go API, it spawned 100 Goroutines, which take up almost zero memory and require zero OS-level context switching. Go just calmly distributed those Goroutines across 2.5 CPU cores, did the math, and handed back the responses in under 1.5 seconds.

3. The Memory Tax
When Java had to juggle those 100 threads and the massive influx of nested JSON objects, its memory footprint tripled from its idle state, peaking at 658 MiB. The Garbage Collector was likely working overtime just to keep the application alive.

Go, handling the exact same workload, peaked at a microscopic 194 MiB.
